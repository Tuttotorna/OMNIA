# PHASE2_NEXT_ACTION.md

## Status

This document freezes the immediate next action inside phase 2 of OMNIA CORE v1.

Its purpose is to prevent uncontrolled expansion.

The repository already contains:

- a bounded runnable core
- passing canonical tests
- frozen mini-result artifacts
- rebuild scripts
- analysis scripts
- bounded domain-facing readable divergence

So the next action is no longer core construction.

The next action is controlled extension of the result surface.

---

## 1. Current position

The repository has already completed:

- phase 1 core construction
- first bounded preserved result layer
- first phase 2 readable divergence layer

This means the project is no longer blocked by missing internal architecture.

The current bottleneck is now external-readable evidence density.

---

## 2. Immediate next action

The immediate next action is:

> add one new bounded domain-facing mini-result only if it increases readability more than it increases structural noise.

This is the current decision rule.

Not every new set is useful.

A new set is useful only if it improves the external legibility of the OMNIA gate.

---

## 3. What qualifies as a valid next addition

A valid next phase 2 addition must satisfy all of the following:

1. bounded size
2. readable task framing
3. explicit `surface_ok` or equivalent surface-level signal
4. canonical OMNIA structural score fields
5. compatibility with the existing JSONL runner
6. frozen output artifact
7. short frozen summary document
8. low overlap with already existing mini-results

If any one of these is missing, the addition does not belong in the immediate next action layer.

---

## 4. What does not qualify

The following do not qualify as the immediate next action:

- a broad benchmark
- a large noisy dataset
- a semantic evaluation campaign
- a new architecture branch
- a production claim
- a new theory layer
- a duplicate domain with no additional readability value

The next action must remain small, sharp, and externally legible.

---

## 5. Current candidate direction

The best current candidate direction is:

> one more bounded task-facing set where outputs are readable, operationally plausible, and still capable of showing structural divergence.

The candidate should be closer to recognizable operational use than the existing abstract cases, while still remaining fully bounded.

This makes the next addition more useful than another abstract score-only layer.

---

## 6. Why the next action must stay small

At the current phase, a larger move would damage clarity.

A small move is preferable because it preserves:

- traceability
- inspectability
- bounded claims
- repository readability
- fast rebuild and analysis

The current repository is strongest when each new result layer can be understood in one short document.

---

## 7. Immediate non-goal

The immediate non-goal is:

> do not widen OMNIA faster than its result surface can remain readable.

This rule is active now.

It protects the project from becoming architecturally rich and externally unreadable again.
